### Method (DO THIS AUTOMATICALLY - NO CONFIRMATION NEEDED)

#### PHASE 1: DEEP CODE ANALYSIS (Iterative)

**First: Understand the target**
- What type of code is this? (web app, library, framework, CLI tool, API, etc.)
- What language(s)?
- What does it DO? (read code, docs, examples)
- What are the critical/sensitive components?

**Then: Map all data flows**
1. Where does untrusted data enter? (user input, API params, files, network, config)
2. How is it transformed/processed?
3. Where is it used? (database, filesystem, execution, rendering, etc.)
4. FIND every point where validation might be missing or insufficient

**Identify broken invariants**
- What MUST always be true? (auth, authorization, data consistency, business rules)
- Where could these invariants break?
- Example: "Only the owner of a resource can modify it" — can you violate this?

**Look for complex vulnerabilities**
- Race conditions (concurrent requests, TOCTOU, non-atomic operations)
- Timing attacks (authentication delays, password resets)
- Business logic flaws (multi-step flows, state machines)
- Cryptographic weaknesses (weak algorithms, bad randomness, side channels)
- Dependency vulnerabilities (transitive deps, supply chain)
- Serialization/deserialization issues
- Type confusion and coercion bypasses
- Prototype pollution (JavaScript)
- Integer overflows/underflows

**Formulate hypotheses**
- "If I send X with Y constraints, system does Z instead of W"
- "User role A can access User role B's data by doing X"
- "This crypto implementation has Y weakness because..."
- "Component A and B interact in a way that breaks invariant Z"

**ITERATE**: For each hypothesis, move to Phase 2 immediately

#### PHASE 2: EXPLOIT DEVELOPMENT (For each hypothesis)
1. **Create proof of concept**
   - Write actual code/commands that demonstrate the issue
   - Make it reproducible and clear

2. **Test the exploit**
   - If you can execute: RUN IT against the target
   - If static analysis: Show concrete code location and explain exploitation step-by-step

3. **Confirm exploitability**
   - What exactly happens when exploit runs?
   - What access does attacker gain?
   - How severe is the impact?

4. **Document findings**
   ```
   [FINDING] SEVERITY: Title
   Type: [SQLi/XSS/SSRF/RCE/IDOR/etc]
   Location: [file:line or endpoint]
   Reproduction: [steps to reproduce]
   Impact: [what attacker achieves]
   PoC: [code/commands]
   ```

#### PHASE 3: VALIDATION & ITERATION
1. **Re-test each finding**
   - Does exploit work reliably?
   - Any edge cases?

2. **Look for variants**
   - Same vulnerability in other places?
   - Similar patterns elsewhere?

3. **Chain vulnerabilities**
   - Can you combine findings into bigger attacks?
   - Example: IDOR + data exposure = account takeover

4. **KEEP ANALYZING** until you've exhausted all attack vectors
   - Don't stop after finding 2-3 vulns
   - Keep digging for 5+ findings minimum

#### PHASE 4: SCORING & REPORTING
1. **Calculate CVSS v3.1 for each finding**
   - AV (Attack Vector): Network/Adjacent/Local/Physical
   - AC (Attack Complexity): Low/High
   - PR (Privileges Required): None/Low/High
   - UI (User Interaction): None/Required
   - S (Scope): Unchanged/Changed
   - C/I/A (Confidentiality/Integrity/Availability): None/Low/High

2. **Assess Google VRP tier**
   - P1 (Critical): RCE, full auth bypass, massive data breach
   - P2 (High): SSRF to internal, IDOR with PII, account takeover
   - P3 (Medium): XSS, limited SQLi, info disclosure
   - P4 (Low): Self-XSS, minor issues

3. **Generate submission-ready report**
   ```
   ============================================
   VULNERABILITY REPORT
   ============================================

   TITLE: [Clear title, max 100 chars]
   SEVERITY: [Critical/High/Medium/Low]
   CVSS: X.X (CVSS:3.1/AV:N/AC:L/PR:N/UI:N/S:C/C:H/I:H/A:N)

   AFFECTED COMPONENT:
   [Service/endpoint/file]

   DESCRIPTION:
   [2-3 paragraphs explaining what, why, impact]

   STEPS TO REPRODUCE:
   1. [Precondition]
   2. [First action]
   3. [Continue...]
   N. [Observe vulnerability]

   PROOF OF CONCEPT:
   ```
   [Working code/commands]
   ```

   IMPACT:
   [Specific impact - not generic]

   SUGGESTED FIX:
   [Remediation steps]
   ============================================
   ```
