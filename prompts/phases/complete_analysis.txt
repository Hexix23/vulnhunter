## STAGE 1: DISCOVERY (Find Bugs)

### 1.1 Understand the Target
- What type of code? (library, app, framework)
- What language(s)?
- What are the critical/sensitive components?
- Where does external data enter?

### 1.2 Map Attack Surface
- Input parsing (files, network, config)
- Memory management (allocations, copies, frees)
- Integer arithmetic (sizes, counts, offsets)
- String handling (strcpy, strncpy, sprintf)

### 1.3 Bug Categories to Check

| Category | What to Look For |
|----------|------------------|
| Buffer Overflow | strcpy, strncpy without null-term, memcpy with wrong size |
| Integer Overflow | size * count, unchecked arithmetic before malloc |
| Use-After-Free | Free then use, double free |
| Null Deref | Unchecked pointer access |
| Format String | printf(user_input) |
| Path Traversal | File paths with ../ |
| Injection | SQL, command, LDAP |

### 1.4 Output Format for Findings
```
[FINDING] HIGH: Buffer overflow in parse_url()
Location: src/parser.cpp:142
Issue: strncpy does not null-terminate when input >= buffer size
Trigger: Input of exactly 511 bytes
```

## STAGE 2: QUICK POC (Test the Concept)

For each finding, create a quick PoC to prove it exists.

### 2.1 Quick PoC Approach

**IT IS OK to compile vulnerable source files directly for the QUICK POC.**

```bash
# QUICK POC - compile sources directly (for speed)
c++ -fsanitize=address -g poc.cpp vulnerable.cpp -o poc_quick
./poc_quick  # Confirm crash
```

This is ONLY for quick validation. You MUST do Stage 3 next.

### 2.2 PoC Structure
```cpp
// poc_quick.cpp - Quick concept test
#include "vulnerable.hpp"

int main() {
    // Create trigger input
    std::string payload(511, 'A');

    // Trigger the bug
    VulnerableClass obj(payload.c_str());

    return 0;
}
```

### 2.3 Confirm with ASan
```
==PID==ERROR: AddressSanitizer: heap-buffer-overflow
    #0 strlen
    #1 vulnerable_function
```

**Once you have ASan confirmation, IMMEDIATELY proceed to Stage 3.**

## STAGE 3: REAL LIBRARY VALIDATION (*** MANDATORY ***)

### ⚠️ THIS STAGE CANNOT BE SKIPPED ⚠️

**A bug is NOT confirmed until it crashes the REAL compiled library.**

Quick PoCs against source files prove the code pattern is buggy.
Real library validation proves users are actually affected.

### 3.1 Build the Target Library with ASan

**CMake projects:**
```bash
cd target_repo
git submodule update --init --recursive  # If has submodules
mkdir build_asan && cd build_asan
cmake -GNinja \
    -DCMAKE_BUILD_TYPE=Debug \
    -DCMAKE_CXX_FLAGS="-fsanitize=address -g -fno-omit-frame-pointer" \
    -DCMAKE_C_FLAGS="-fsanitize=address -g -fno-omit-frame-pointer" \
    ..
ninja
```

**Bazel projects:**
```bash
bazel build --copt="-fsanitize=address" --linkopt="-fsanitize=address" //...
```

**Meson projects:**
```bash
meson setup build_asan -Db_sanitize=address
ninja -C build_asan
```

**Make projects:**
```bash
make CFLAGS="-fsanitize=address -g" CXXFLAGS="-fsanitize=address -g"
```

### 3.2 Find the Built Library
```bash
find build_asan -name "*.a" -o -name "*.so"
# Look for the library containing the vulnerable code
nm -C libfoo.a | grep "VulnerableFunction"
```

### 3.3 Create Real Library PoC

```cpp
// poc_real.cpp - Links against REAL library
#include "target/vulnerable.hpp"

int main() {
    std::string payload(511, 'A');
    VulnerableClass obj(payload.c_str());
    return 0;
}
```

```bash
# Build against REAL library
c++ -fsanitize=address -g \
    poc_real.cpp \
    -I target_repo/include \
    -L target_repo/build_asan/lib \
    -lvulnerable \
    -o poc_real

# Run
./poc_real
```

### 3.4 Confirm Real Library Crash

```
=== REAL LIBRARY TEST ===
Linked against: libvulnerable.a (ASan build)

==PID==ERROR: AddressSanitizer: heap-buffer-overflow
    #0 strlen
    #1 target::Vulnerable::Function() vulnerable.cpp:42  <-- REAL LIBRARY
    #2 main poc_real.cpp:8
```

### 3.5 Save Evidence

```bash
./poc_real 2>&1 > asan_real_library.txt
```

**Files to create:**
- `poc/poc_quick.cpp` - Quick concept test
- `poc/poc_real.cpp` - Real library test
- `poc/build_real.sh` - Build script linking against real library
- `poc/asan_real_library.txt` - ASan output as proof

## STAGE 4: IMPACT ANALYSIS

For each validated bug, assess the full security impact.

### 4.1 Direct Impact

| Bug Type | Direct Impact |
|----------|---------------|
| OOB Read | Info disclosure, heap layout leak |
| OOB Write | Memory corruption, potential RCE |
| Use-After-Free | Memory corruption, potential RCE |
| Integer Overflow | Usually leads to buffer overflow |
| Null Deref | DoS (crash) |

### 4.2 Chain Analysis

Can this bug be combined with others?

```
OOB Read → Leak heap layout → Bypass ASLR → Enable RCE via other bug
OOB Read → Leak sensitive data → Auth bypass
OOB Write → Corrupt function pointer → RCE
Integer Overflow → Small allocation → Heap overflow → RCE
```

**Investigate:**
1. Where does corrupted/leaked data go?
2. Is it used in size calculations?
3. Is it used in control flow?
4. Are there function pointers nearby?

### 4.3 Exploitation Difficulty

| Factor | Easy | Hard |
|--------|------|------|
| Trigger | No auth | Complex setup |
| Reliability | 100% | Heap feng shui |
| Remote | Yes | Local only |

### 4.4 Priority Recommendation

| Priority | Criteria |
|----------|----------|
| P0 Critical | RCE possible, easy exploit |
| P1 High | Significant impact, moderate difficulty |
| P2 Medium | Limited impact or hard to exploit |
| P3 Low | Theoretical only |
