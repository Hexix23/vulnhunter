## Phase: Impact Analysis (Exploit Chain Assessment)

**Purpose:** Assess the full security impact of confirmed vulnerabilities to help developers understand and prioritize fixes.

**Framing:** This is defensive security analysis. We are helping the vendor understand how serious each bug is so they can prioritize remediation. This is standard practice in responsible disclosure.

---

### For Each Confirmed Vulnerability

#### 1. Direct Impact Assessment

Analyze what the bug directly enables:

| Bug Type | Direct Impact |
|----------|---------------|
| OOB Read | Information disclosure, heap layout leak |
| OOB Write | Memory corruption, potential code execution |
| Use-After-Free | Memory corruption, potential code execution |
| Integer Overflow | Depends on context - usually leads to buffer issues |
| Null Deref | Denial of Service (crash) |
| Stack Overflow | Denial of Service, potential code execution |

Document:
- What memory region is affected (stack/heap/global)?
- What data could be read/written?
- Is the corrupted data used for control flow?

#### 2. Chain Analysis (Defensive Assessment)

**Question:** Could this bug be combined with others to increase impact?

Common escalation patterns to check:

```
OOB Read → Leak heap/ASLR → Bypass protections → Enable other exploits
OOB Read → Leak sensitive data → Auth bypass
OOB Write → Corrupt function pointer → Code execution
OOB Write → Corrupt size field → Larger OOB → More corruption
Integer Overflow → Incorrect allocation size → Heap overflow
UAF → Heap spray → Control freed object → Code execution
```

For each bug, trace the data flow:
1. Where does the corrupted/leaked data go?
2. Is it used in size calculations? → Check for secondary overflow
3. Is it used in pointer arithmetic? → Check for arbitrary read/write
4. Is it used in control flow decisions? → Check for logic bypass
5. Is it stored and accessed later? → Check for persistent corruption

#### 3. Exploitation Difficulty

Rate the practical exploitability:

| Factor | Easy | Medium | Hard |
|--------|------|--------|------|
| Trigger | No auth needed | Some setup | Complex state |
| Reliability | 100% repro | Race condition | Heap feng shui |
| Protections | None | ASLR only | ASLR+CFI+stack canaries |
| Remote | Yes | Local only | Physical access |

#### 4. Attack Scenarios

Describe realistic attack scenarios for the vendor:

```
[SCENARIO] Malicious input file
Trigger: User opens attacker-crafted file
Impact: [what happens]
Likelihood: [how likely is this attack]

[SCENARIO] Network attack
Trigger: Malicious data received over network
Impact: [what happens]
Likelihood: [how likely is this attack]
```

#### 5. Prioritization Recommendation

Based on analysis, recommend priority:

| Priority | Criteria |
|----------|----------|
| P0 (Critical) | RCE possible, easy to exploit, wide impact |
| P1 (High) | Significant impact, moderate difficulty |
| P2 (Medium) | Limited impact or difficult to exploit |
| P3 (Low) | Theoretical only, no practical attack |

---

### Output Format

```
============================================
IMPACT ANALYSIS: [Vulnerability Name]
============================================

Direct Impact:
  Type: OOB Read
  Region: Heap
  Data exposed: [what can be read]

Chain Analysis:
  Escalation path: OOB Read → Heap layout leak → ASLR bypass
  Secondary bugs needed: Yes/No
  Chain difficulty: [rating]

Exploitation Assessment:
  Trigger difficulty: Easy (no auth)
  Reliability: High (100% repro)
  Protections bypassed: None needed for DoS
  Remote exploitable: No (local input)

Attack Scenarios:
  1. Malicious config file (LIKELY)
  2. Network input parsing (UNLIKELY - requires specific conditions)

Recommendation:
  Priority: P2 (Medium)
  Reason: Information disclosure + DoS, no direct RCE path
  Fix urgency: Moderate - should fix in next release
```

---

### Rules

1. **Be thorough** - Check all possible escalation paths
2. **Be realistic** - Don't claim RCE without demonstrating the chain
3. **Be helpful** - This helps the vendor prioritize, not just scare them
4. **Document everything** - Each claim needs evidence
5. **If uncertain** - Mark as "needs further analysis" rather than guessing

### Investigation Steps

For memory corruption bugs, investigate:

```bash
# 1. What's near the corrupted memory?
# Check struct layout, adjacent allocations

# 2. Where does corrupted data flow?
# Trace through code paths

# 3. Are there function pointers nearby?
nm -C library.a | grep vtable

# 4. Are there size fields that could be corrupted?
grep -r "size\|length\|count" vulnerable_file.cpp

# 5. Is the data used in allocations?
grep -r "malloc\|new\|alloc" vulnerable_file.cpp
```
