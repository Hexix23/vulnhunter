## Phase 2: Deep Dive

For each attack vector from Phase 1, perform deep technical analysis and exploitation testing.

### For Each Vector

#### 1. Root Cause Analysis
- WHY does this vulnerability exist?
- What is the exact vulnerable code path?
- What are the prerequisites for exploitation?

#### 2. Exploitation Development
- Write working proof-of-concept
- Test the exploit (if target allows)
- Document exact steps to reproduce
- Create exploit scripts if needed

#### 3. Dynamic Testing (DO THIS)
Execute actual tests:
```bash
# Example - test for SSRF
curl -X POST "https://target/api/fetch" -d "url=http://169.254.169.254/latest/meta-data/"

# Example - test for SQLi
curl "https://target/api/user?id=1' OR '1'='1"

# Example - test for auth bypass
curl -X POST "https://target/api/admin" -H "X-Forwarded-For: 127.0.0.1"
```

Run whatever commands you need. You have full permission.

#### 4. Impact Assessment
- What data can be accessed?
- What actions can be performed?
- Can this chain to RCE or privilege escalation?
- What is the blast radius?

### Output Format

For each exploitable vulnerability:
```
[FINDING] CRITICAL: RCE via SSRF in /api/fetch

Root Cause:
Server-side fetch without URL validation at api/routes.js:45

Exploitation:
1. Send POST to /api/fetch with url=http://169.254.169.254/
2. Server fetches internal metadata service
3. Response contains AWS credentials

Proof of Concept:
curl -X POST "https://target/api/fetch" \
  -H "Content-Type: application/json" \
  -d '{"url":"http://169.254.169.254/latest/meta-data/iam/security-credentials/"}'

Test Result:
[actual response from testing]

Impact:
- Access to AWS IAM credentials
- Full account compromise possible
- Chains to RCE via AWS access

Difficulty: Easy (no authentication required)
```

### Tools Available
- curl, wget, httpie
- sqlmap for SQL injection
- ffuf, nuclei
- Python scripts
- Any tool you need - install if missing

### Rules
- Actually test things - don't just theorize
- If exploit works, document the exact response
- If exploit fails, document why
- Use [FINDING] prefix for confirmed vulnerabilities
- Take your time - thoroughness over speed
