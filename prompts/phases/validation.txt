## Phase 3: Validation

Validate all findings from Phase 2. Confirm exploitability and assess for Google VRP submission.

**CRITICAL: All memory corruption bugs MUST be validated against the REAL compiled library, not isolated source files.**

### For Each Finding

#### 0. MANDATORY: Real Library Validation (Memory Bugs)

For any memory corruption (buffer overflow, OOB, use-after-free, etc.):

1. **Build the target project** with its real build system (CMake, Bazel, Meson, etc.)
2. **Enable sanitizers** in the build: `-fsanitize=address -g`
3. **Link PoC against the compiled library** (.a or .so), NOT source files
4. **Run and capture ASan output** proving the bug exists in the real library

Example for CMake projects:
```bash
git submodule update --init --recursive
mkdir build_asan && cd build_asan
cmake -DCMAKE_CXX_FLAGS="-fsanitize=address -g" ..
make -j$(nproc)
# Then link PoC against build_asan/lib*.a
```

**DO NOT** compile vulnerable .cpp files directly into a harness - this proves nothing about the real product.

**DO** link against the actual library artifacts that users compile and use.

#### 1. Scope Verification
Check against Google VRP rules (bughunters.google.com):
- Is the target service in scope?
- Is this vulnerability type accepted?
- Are there any exclusions that apply?

**If out of scope**: Mark as OUT_OF_SCOPE and explain why.

#### 2. Reproducibility Check
- Re-run the exploit
- Confirm it still works
- Test from different conditions
- Document any variations

#### 3. CVSS Scoring
Calculate CVSS v3.1:
- Attack Vector (AV): Network/Adjacent/Local/Physical
- Attack Complexity (AC): Low/High
- Privileges Required (PR): None/Low/High
- User Interaction (UI): None/Required
- Scope (S): Unchanged/Changed
- Confidentiality (C): None/Low/High
- Integrity (I): None/Low/High
- Availability (A): None/Low/High

Example: CVSS:3.1/AV:N/AC:L/PR:N/UI:N/S:C/C:H/I:H/A:N = 10.0

#### 4. Google VRP Tier
- **P1 (Critical)**: RCE, full auth bypass, data breach affecting many users
- **P2 (High)**: SSRF to internal services, IDOR exposing PII, account takeover
- **P3 (Medium)**: XSS, limited SQLi, info disclosure
- **P4 (Low)**: Self-XSS, minor issues

#### 5. Duplicate Check
- Is this a known issue?
- Similar to recent public reports?
- Variant of a common vulnerability?
